# Техническое задание: LNN-based Low-Light Video Enhancement

**Версия:** 0.1  
**Дата:** 2026-06-05  
**Статус:** Draft  

---

## 1. Цель проекта

Исследовать применимость Liquid Neural Networks (LNN) как компонента для моделирования временно́й динамики освещения в задаче улучшения видео при слабом освещении (Low-Light Video Enhancement, LLVE).

**Основная гипотеза:**  
Непрерывная ODE-динамика LNN лучше моделирует физику изменения illumination map во времени, чем дискретные рекуррентные сети (LSTM/GRU), что приводит к улучшению метрик качества и временно́й консистентности.

**Вторичная гипотеза:**  
Обученный LNN-модуль может быть переиспользован как компактная суррогатная модель динамики освещения для real-time рендеринга (игры, XR-приложения).

---

## 2. Контекст и мотивация

### 2.1 Проблема
Существующие методы LLVE (RetinexFormer, NAFNet и др.) обрабатывают каждый кадр независимо или используют дискретные рекуррентные сети для временно́й связности. Изменение освещения — физически непрерывный процесс, описываемый ODE. LNN (и их closed-form вариант CfC) решают ODE явно, что теоретически соответствует природе задачи.

### 2.2 Новизна
- Применение LNN/CfC к LLVE не описано в литературе (по состоянию на июнь 2026)
- Интеграция LNN в Retinex-декомпозицию (между illumination estimator и damage restorer) — новая архитектурная идея
- Сравнение LNN vs LSTM/GRU как темпоральных компонентов в одинаковом пайплайне — измеримый вклад

### 2.3 Смежные работы
- **RetinexFormer** (ICCV 2023, NTIRE 2024 Runner-up, NTIRE 2025/2026 Winner) — базовая архитектура
- **Vid-ODE** (ODE-ConvGRU для видеогенерации) — структурный аналог
- **HSI-CfC** (CNN + CfC для гиперспектральных изображений) — паттерн интеграции
- **TempRetinex** (unsupervised Retinex для видео) — ближайший конкурент по задаче
- **TransGI / Neural Light Transport** — контекст для суррогатного применения

## 3. Архитектура

### 3.1 Основная архитектура: Retinex-LNN

```
Входное видео (кадры t-N ... t)
         │
         ▼

└─────────────────────┘
         │
   illumination_map_t ──────────────────────────┐
         │                                       │
         ▼                                       ▼
┌─────────────────────┐              ┌─────────────────────┐
│  LNN (CfC)          │              │  Reflectance        │
│  Temporal Module    │              │  (передаётся        │
│                     │              │   напрямую)         │
│  Моделирует ODE     │              └─────────────────────┘
│  динамики света     │                        │
└─────────────────────┘                        │
         │                                     │
   illum_enhanced_t                            │
         │                                     │
         └──────────────┬──────────────────────┘
                        ▼
             ┌─────────────────────┐
             │  CNN Decoder        │  ← RetinexFormer Damage Restorer
             │  (RetinexFormer)    │
             └─────────────────────┘
                        │

```


### 3.2 Варианты LNN-модуля

| Вариант | Описание | Когда использовать |
|---|---|---|
| **CfC** (Closed-form Continuous-time) | Аналитическое решение LTC, быстрее | Основной вариант |
| **LTC** (Liquid Time-Constant) | Оригинальный, ODE-solver, медленнее | Ablation study |

### 3.3 Baseline для сравнения

Та же архитектура с заменой LNN-модуля:
- **LSTM** — стандартный baseline
- **GRU** — лёгкий baseline  
- **Без темпорального модуля** — нижняя граница (per-frame RetinexFormer)

- [ ] Обучить, получить метрики
- [ ] Ablation: LTC vs CfC

**Выход:** основной результат исследования

## 11. Расширение: Transformer temporal module (LNN vs Transformer)

**Версия:** 0.1
**Дата:** 2026-07-06
**Статус:** Утверждено — готово к реализации

### 11.1 Цель

Проверить основную гипотезу проекта в её наиболее амбициозной форме: способна ли
существенно более простая LNN (CfC) конкурировать по качеству и temporal
consistency с self-attention temporal-модулем (Transformer) при контролируемом
сравнении — тот же backbone, тот же loss, тот же датасет, тот же тренировочный
бюджет, единственная переменная — тип temporal-головы.

Это Путь А из обсуждения (в противовес Пути Б — сравнению с полноценными
published video-restoration трансформерами со своим backbone/весами/рецептом
обучения, что на порядок дороже и не планируется в рамках этого расширения).

### 11.2 Место в существующей архитектуре

Модуль встраивается в уже существующий registry `TEMPORAL_MODULES`
(`models/temporal/__init__.py`) как четвёртый пир к CfC/LSTM/GRU, с идентичным
интерфейсом:

```python
def forward(self, illu_fea_seq, timespans=None) -> tuple[Tensor, Tensor | None]:
    """illu_fea_seq: [B, T, C, H, W] -> (illu_fea_curr, illu_fea_prev_or_None)"""
```

Реализация: spatial pooling (переиспользуется существующий `SpatialPooling` из
`lstm_module.py`) → temporal self-attention над окном T шагов → spatial
broadcast (существующий `SpatialBroadcast`) → тот же residual-gate merge
(`_finalize`), что уже есть у CfC/LSTM/GRU.

### 11.3 Открытые вопросы и принятые решения

| # | Вопрос | Решение | Почему это важно |
|---|---|---|---|
| 1 | Causal vs bidirectional attention | **Causal-маска** (нижнетреугольная, шаг t не видит t+1..T-1) | CfC/LSTM/GRU строго causal по построению (рекуррентность). Bidirectional attention дал бы Transformer доступ к будущим кадрам окна — информационное преимущество, не связанное с архитектурой, делающее сравнение нечестным. |
| 2 | Бюджет параметров | **Вариант (а): равный бюджет**, ~28k параметров (как у CfC), т.е. малая размерность attention (небольшой `d_model`, 1-2 головы) | Гипотеза — "простая LNN конкурентна с Transformer". При равном бюджете это прямой тест исходного утверждения. Вариант (б) — Transformer в типичном для задачи размере (на порядок больше) — отдельный ablation *после* (а), если результат (а) заслуживает продолжения. |
| 3 | Positional encoding | **Learned positional embedding** по индексу шага в окне (T=5 → 5 позиций, тривиально по объёму) | Attention не видит порядок кадров по умолчанию (в отличие от RNN, где порядок встроен в рекуррентность) — без этого модуль не будет знать, какой кадр за каким идёт. |
| 4 | Backbone init parity (см. 11.4) | **Общая случайная инициализация backbone**, зафиксированная из уже выполненного CfC-прогона | Backbone заморожен во всех арках сравнения — если инициализация будет каждый раз новой случайной, это добавит неучтённый шум, из-за которого разница между temporal-модулями станет неотличима от разницы в случайном стартовом состоянии backbone. |

### 11.4 Сохранение прогресса предыдущих циклов обучения

Прогресс обучения CfC-головы (её веса) **нельзя** перенести в LSTM/GRU/Transformer
напрямую — это разные архитектуры с разным пространством параметров, ничего не
поделать. Но кое-что реально и нужно сохранить:

1. **CfC-прогон** (уже идущий, эпоха 2 частично пройдена) — продолжается через
   уже существующий `--resume experiments/checkpoints/cfc_bvirlv/latest.pth`.
   Прогресс полностью сохраняется, ничего не перезапускается с нуля.

2. **Backbone-инициализация для новых прогонов (LSTM/GRU/Transformer)** —
   поскольку backbone заморожен и никогда не обновляется градиентом, веса
   `estimator`+`denoiser` в `latest.pth` от CfC-прогона буквально равны их
   случайной инициализации при создании модели. Извлекаем эти веса один раз и
   переиспользуем как общую стартовую точку для всех новых арок сравнения —
   это не экономия вычислений (backbone и так не обучался), а требование
   методологической валидности: без этого разный random seed backbone стал бы
   неучтённым источником шума в сравнении между temporal-модулями.

   **Нужна небольшая доработка кода:** функция частичной загрузки чекпоинта
   (`load_backbone_from_checkpoint(model, ckpt_path)`), копирующая только ключи
   `estimator.*` и `denoiser.*` из state_dict — так как остальная часть
   state_dict (temporal-модуль) несовместима между архитектурами по форме
   тензоров.

### 11.5 Ожидаемые результаты

**Минимальный результат:** CfC не хуже Transformer по PSNR/tOF при равном
бюджете параметров (11.3, п.2) — подтверждает, что архитектурная простота не
стоит качества.

**Целевой результат:** CfC превосходит Transformer по tOF (temporal
consistency) при равном или меньшем бюджете — прямое подтверждение основной
гипотезы проекта в её наиболее сильной форме.

**Дальнейший шаг (не входит в это расширение):** если целевой результат
достигнут — рассмотреть Путь Б (сравнение с published video-restoration
трансформерами) как отдельный, отдельно спланированный подпроект.

### 11.6 Оценка объёма работ

Новый файл `models/temporal/transformer_module.py` (~80-100 строк, по образцу
`lstm_module.py`) + функция `load_backbone_from_checkpoint` в `train.py` +
новый конфиг `configs/retinex_transformer_bvirlv.yaml` + тест в
`smoke_test.py` (по аналогии с существующими 5a/5b/5c). Инженерно сопоставимо
с объёмом прошлого bug-фикса (train/temporal=0.000, коммит `e0ff7dc`).
Основная стоимость — не код, а тренировочный бюджет: ещё три полных прогона
(LSTM, GRU, Transformer) по той же схеме 2.5ч/сессия, что и текущий CfC.
